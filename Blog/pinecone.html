Pinecone - Networking in Unity

  • Posted on June 4, 2022

This post is going to introduce my final year project (Pinecone), a networking library made for the Unity engine. It will discuss how I implemented it and what I learned from creating such a project.

The Idea

As part of a university module called "Connected Networking Applications", I learned how to create a basic server and client that could send data between each other. This inspired me to continue working with multiplayer games, something that I had approached once for another module. I had never written any backend parts of multiplayers games, only utilized existing libraries. I therefore decided that I wanted to attempt creating a collection of code that could help me create a multiplayer game.

I thought quite hard about it. Instead of making a library for one type of game, why not approach a library that could be used for multiple games. If I wanted to ever make my own multiplayer games, I could use the library that I had made for this module. This is where Pinecone started.

Why Unity?

Once I had the idea, I looked into which environment that I wanted to start a networking library in. There were 3 options:

  • C#/Unity
  • C++/UE4
  • C++/Custom Engine
I nautrally thought of Unity. The main reason why was simple: UE4 already has a networking solution and C++ plus any framework would require myself to create an engine first. The latter seemed very out of scope so I decided to pursue with Unity. The other reason why was also just that Unity did not have their own solution at the time. Netcode for Gameobjects got announced after I had handed in my proposal. Using C# would also allow me to focus on the overall tool within the engine as .NET comes with a bunch of helper classes for networking. Finally, I also had prior experience using .NET from the module I had already taken.

The Plan

Now that the idea had been settled, I started researching a bunch into how I would create this massive project. Although, there are resources on creating a networked application, it is difficult to find much specifically related to games. A great resource I found during my researching phase is GameNetworkingResources. It is a curated list of game network programming resources that is frequently updated.

With the research acquired and me looking into how other libraries for Unity works, I was able to come up with a plan for abstracting the Transport layer of the networking library so that it was replaceable. This allows for custom implementations beyond what the library includes by default. For instance, if developers wished to use Steam networking, they would be unable to use the Transport layer that I included. The backend system however would still work.
